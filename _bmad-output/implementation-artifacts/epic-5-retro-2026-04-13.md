## What Went Well

### The hardening-only scope stayed disciplined
Epic 5 did not sprawl into new runtime features. Each story tightened one aspect of the Phase 2 surface: contracts first, transport parity second, docs and troubleshooting third, and rollout regression coverage fourth.

### Shared contract boundaries held under pressure
Story 5.1 extended typed error handling and normalized response patterns without inventing parallel schema systems or leaking raw internal exceptions. That is the right outcome for a hardening epic.

### Transport-neutral behavior was enforced rather than assumed
Story 5.2 and Story 5.4 both verified parity across `stdio` and Streamable HTTP with real bootstrap and smoke-level checks. This reduced the chance of configuration- or transport-specific drift sneaking in after the feature epics were done.

### Documentation was treated as part of the product surface
Story 5.3 and the later review-fix pass corrected setup, integration, and troubleshooting guidance to match the actual server behavior instead of letting docs lag behind the implementation.

### Review findings produced targeted improvements instead of churn
The final review found real issues, not cosmetic ones: stale error-category references in integration guides, hand-maintained release-test partitions, and workflow assertions that were still too formatting-sensitive. Those issues were fixed narrowly and validated quickly.

---

## What Didn't Go Well

### Docs truthfulness still lagged behind the contract in at least one spot
Even after adding the shared `internal` error category in Story 5.1, the Home Assistant and `n8n` integration guides continued to describe only four shared categories until the review-fix pass caught the drift. That means the docs regression net was not yet complete when the epic was first marked done.

### Release-validation tests drifted toward implementation-coupled checks
The initial Story 5.4 release-helper tests improved coverage, but they still relied on hand-maintained tool partitions and indentation-sensitive workflow parsing. The test intent was right; the first implementation was still too brittle.

### Sprint-tracker hygiene was corrected late again
Epic 5 reached functional completion before all tracker fields were updated consistently. `current_completed_story`, `next_story_hint`, `epic-5`, and `epic-5-retrospective` needed follow-up corrections, and the retrospective markdown file itself was missing until this pass.

### Epic closure still depends too much on manual artifact sync
The code, docs, story files, sprint tracker, and retrospective artifacts are all individually understandable, but closing an epic still requires someone to remember to update each one explicitly. That is operational friction.

---

## Lessons Learned

1. **Hardening epics need the same rigor as feature epics.** A no-new-features epic can still ship real defects if the docs, release checks, and tracker artifacts are treated as secondary.

2. **The docs regression suite is the right home for public-surface completeness checks.** Exact tool-surface and error-category truth belongs in docs-facing regression tests, not duplicated inside release-helper tests with hand-maintained partitions.

3. **Release semantics tests should assert meaning, not YAML formatting.** Workflow checks should verify tag-triggered release behavior and required commands, not depend on indentation shape or fragile regex block extraction.

4. **Tracker hygiene is part of epic completion, not post-epic cleanup.** If an epic is done, the tracker, `current_completed_story`, next-step hints, retrospective marker, and retrospective artifact should all reflect that immediately.

5. **Review-driven fix passes are valuable when they stay narrow and evidence-based.** The Epic 5 follow-up fixes improved truthfulness and test durability without reopening Phase 2 scope.
