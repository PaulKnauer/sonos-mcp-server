## What Went Well

### Story scoping stayed disciplined
Epic 3 preserved strong separation between verifier logic, startup validation, server wiring, and coverage expansion. That made the work easier to reason about and kept changes coherent under review.

### Review loops materially improved the implementation
The best example is Story 3.3. Review found real startup/runtime mismatches around discovered JWKS propagation, audience requirements, discovery URL construction, malformed resource URLs, and CA-bundle diagnostics. The re-review cycle corrected them before the epic closed.

### Boundary discipline from Epic 2 held under higher complexity
The auth concerns stayed in `auth/`, `config/validation.py`, and `server.py`. The implementation did not leak into `tools/`, domain logic, or transports beyond the intended server boundary.

### Quality depth expanded with the feature
Epic 3 did not stop at mocked happy-path coverage. By Story 3.4, the verifier suite exercised real RS256 tokens, JWKS lookup by `kid`, rotation refresh behavior, and exact FastMCP verifier typing expectations without external provider dependencies.

### Validation ownership became clearer in practice
Epic 2 flagged soft ownership boundaries between model validation, preflight, and runtime auth. Epic 3 improved this materially:
- model validation enforces config shape
- preflight owns startup diagnostics and connectivity checks
- verifier owns runtime token decisions

---

## What Didn’t Go Well

### Tracker hygiene drifted again
The optional-auth sprint tracker still needed manual corrections during close-out. This is now a recurring workflow weakness across epics rather than a one-off slip.

### Story 3.3 needed multiple review passes to reach the correct runtime shape
That is not a failure by itself, but it signals that startup/runtime interaction points are where assumptions are most likely to be wrong. This area benefits from aggressive adversarial review and stronger up-front checklists.

### Documentation lag is now the main product risk
Story 3.3 exposed a broken troubleshooting/docs reference during review. The implementation is ahead of the operator-facing guidance, which means Epic 4 is not optional polish; it is the release-readiness layer.

### Some quality signals still rely on manual synthesis
The epic produced strong evidence in story artifacts, but there is still manual effort required to keep story status, tracker state, and retrospective continuity aligned.

---

## Lessons Learned

1. Auth work scales well when verifier behavior, startup validation, server wiring, and coverage are kept in separate story slices with explicit ownership.
2. OIDC startup/runtime seams are the highest-risk area; they deserve adversarial review by default.
3. Disabled-auth and stdio protections remain first-class acceptance criteria whenever auth grows, and Epic 3 successfully proved that this must remain intentional.
4. Documentation accuracy is part of feature completeness. Once runtime behavior becomes stable, docs mismatches become the biggest remaining source of user pain.
5. Tracker correctness is still an engineering-quality issue, not admin overhead.
