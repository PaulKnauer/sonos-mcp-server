## What Went Well

### Documentation became implementation-sensitive

Story 4.1 treated docs as product surface, not cleanup. The auth guide was cross-checked against `models.py`, `loader.py`, `validation.py`, and `server.py`, including exact env var names, OIDC preflight behavior, error categories, and troubleshooting anchors.

### Scope boundaries held

The epic avoided reopening runtime auth behavior while writing adoption material. Story 4.1 stayed documentation-focused, Story 4.2 handled deployment examples and Helm scaffolding, and Story 4.3 handled release validation.

### Review feedback found real release risks

Review found meaningful issues before close-out:

- Story 4.1 corrected docs around HTTP host defaults, Helm guidance, Claude Desktop limitations, OIDC discovery behavior, resource URL semantics, missing audience troubleshooting, model-validator error text, and home-network exposure requirements.
- Story 4.2 corrected Helm rollout behavior, invalid CA-bundle rendering, Docker Compose `SSL_CERT_FILE` pass-through, stale setup docs, Authelia guidance, Helm CA-bundle smoke assertions, and tracker state.
- Story 4.3 corrected release docs so docs tests are actually run and hardened docs regression tests to compare against shipped config surfaces.

Charlie (Senior Dev): "The review loop did its job. It found the exact kind of mismatch operators would hit first."

Paige (Technical Writer): "The useful shift was treating every prose claim as testable against code, chart values, or an actual setup path."

### Provider-free validation stayed intact

The team resisted adding OIDC provider dependencies to CI. Static auth became the release-facing smoke path, while OIDC behavior remains covered through verifier, preflight, server-wiring, and docs checks.

### Epic 3 lessons were applied

Epic 3 warned that docs accuracy, deployment examples, and release validation were the main remaining risks. Epic 4 directly addressed those areas and converted several of them into repeatable checks.

## What Did Not Go Well

### Docs drift was more widespread than expected

Story 4.1 and Story 4.2 both uncovered stale statements across setup, operations, Docker, Helm, README, security, and release docs. The issue was systemic: multiple docs surfaces described the old no-auth posture after auth had already become real.

### Tracker hygiene improved late, not early

Epic 3 carried forward a tracker hygiene action item. Story 4.2 still had a review finding where sprint status and story state disagreed. Story 4.3 closed more cleanly, but the improvement happened through review pressure rather than an already-reliable process.

### Deployment examples needed more defensive checks

The Helm CA-bundle and rollout issues showed that example configuration is executable surface area. Chart values, ConfigMap/Secret split, restart behavior, and optional mount rendering all needed the same quality bar as runtime code.

### Release validation was initially underspecified in docs

Story 4.3 found that release documentation claimed validation confidence without explicitly running docs regression tests. The release checklist needed to name the checks that protect the operator-facing surfaces.

---

## Lessons Learned

1. Operator-facing docs for security features must be verified against code, chart templates, and tests. Prose alone is not enough.
2. Deployment examples are production-adjacent code. They need render tests and drift checks, especially when secrets, config maps, CA bundles, and restart behavior are involved.
3. Static auth is the right smoke path for release validation because it exercises the real HTTP auth boundary without external infrastructure.
4. OIDC should remain provider-free in CI; realistic JWT/JWKS unit coverage plus startup-preflight checks provide better stability than standing up an IdP in the pipeline.
5. Tracker synchronization is still a process weak spot. It improves when stories require explicit evidence, but it should not depend on review finding the drift.
6. Claude Desktop and private home-lab HTTP deployment constraints need careful wording because the local stdio path and remote connector path have materially different trust and reachability models.

## Next Work Preparation

No next optional-auth epic is currently defined in the optional-auth tracker. Future optional-auth work can start from a release-maintenance posture rather than a new feature epic.

Recommended preparation before any future auth expansion:

- Keep the static-auth smoke path as the primary HTTP release gate.
- Keep OIDC CI provider-free unless a concrete, stable provider-independent test seam is unavailable.
- Update docs and Helm/Docker examples in the same change set as config-surface changes.
- Re-run docs regression and Helm rendering checks whenever auth env vars, chart values, or startup diagnostics change.
- Do not start RBAC, claims-based authorization, or provider-specific setup automation without a new planning artifact.

## Conclusion

Epic 4 was successful.

It turned optional authentication from a working runtime capability into an operator-ready release surface. The strongest result was not just the new docs or examples; it was converting the risky parts of those docs and examples into regression checks. The most important remaining improvement is workflow hygiene: tracker sync and durable auth ownership guidance should be made harder to forget.

Optional Authentication v0.6.0 is ready from an Epic 4 adoption, deployment, and release-validation perspective.
