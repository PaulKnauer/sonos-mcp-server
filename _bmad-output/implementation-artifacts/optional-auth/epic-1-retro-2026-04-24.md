## What Went Well

### Scope discipline held
Epic 1 stayed tightly bounded to config, preflight, and backward-compatibility work. No story leaked into runtime verifier implementation, docs, or deployment assets.

### Secret handling was treated as a first-class requirement
`SecretStr` masking, no-secret-in-error-message assertions, and explicit follow-up on review findings established good habits early instead of treating secret safety as a late hardening task.

### The config layer became a stable foundation
Story 1.1 put the full auth config surface in place cleanly, and Stories 1.2 and 1.3 could build on it without redesigning names, defaults, or public imports.

### Preflight stayed the single startup gate
Story 1.2 kept `run_preflight()` as the operator-facing validation boundary. That decision simplified the mental model: config problems fail early, before server bootstrap.

### Regression coverage expanded with the feature
The story records show a consistent pattern of increasing tests and targeted assertions rather than relying on informal confidence. That is the right shape for an opt-in security feature.

---

## What Didn't Go Well

### Review findings exposed boundary ambiguity
Story 1.1 needed follow-up to avoid hard-failing stdio OIDC configs too early, which showed that config validation ownership between model validation and preflight was not fully clear at first pass.

### The disabled-auth “strict no-op” promise was harder than it looked
Story 1.3 needed extra care, explicit tests, and later code review scrutiny because “auth is absent” is not the same thing as “the path is structurally unchanged.” This is an architectural sharp edge going into Epic 2.

### Some operator behaviors are documented in architecture but lightly enforced in code
The architecture says stdio+auth should behave as warning-not-error, but the implementation emphasis in Epic 1 focused more on “must not block” than on making that operator experience fully explicit and uniform.

---

## Lessons Learned

1. For auth work, boundary placement matters as much as correctness. Config model, preflight, and server wiring need crisp ownership or review churn follows.
2. “No-op by default” needs proof, not intent. The team should treat disabled-path assertions as a regression contract any time auth wiring changes.
3. Secret-safety checks belong in tests from the first story. Once `SecretStr` and no-leak assertions were in place, later stories had a clearer safety baseline.
4. Brownfield auth work goes smoother when the public config vocabulary is settled early. Story 1.1 reduced downstream churn by locking in names and defaults.
