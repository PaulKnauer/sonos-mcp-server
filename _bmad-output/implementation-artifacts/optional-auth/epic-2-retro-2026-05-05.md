## What Went Well

### Epic 1 follow-through was real, not performative
Epic 1 asked for strict disabled-auth regression coverage and lazy auth construction. Epic 2 delivered both. Story 2.2 explicitly kept stdio auth ignored, moved auth imports behind the HTTP-auth path, and Story 2.3 extended regression checks around the disabled-auth baseline.

### Boundary discipline stayed intact
The story records consistently reinforced the same architecture rule: auth remained a server/transport concern. The implementation did not leak auth logic into `tools/`, `services/`, `adapters/`, or transport runners. That matters because Epic 3 will add more moving parts and this boundary is now proven in the simpler static path.

### Review loops improved the implementation, not just the paperwork
The code review findings on Story 2.1 and Story 2.2 produced concrete hardening:
- non-ASCII tokens now fail closed
- stdio and disabled-auth behavior stayed clean
- IPv6 issuer host handling was corrected
- supplied async HTTP clients and smoke subprocess cleanup were tightened

That is the right shape of review: targeted, evidence-based, and implementation-improving.

### Test coverage grew with the feature surface
This epic did not rely on “the happy path works” confidence. It added verifier unit tests, server-construction tests, smoke auth checks, and later regression coverage for whitespace semantics, wrong auth schemes, and disabled-auth invariants. The feature became easier to trust because coverage expanded at the same time as behavior.

### Scope stayed honest across the three stories
Story 2.1 stayed narrowly about verifier construction. Story 2.2 handled server wiring and HTTP behavior. Story 2.3 hardened the regression matrix. That separation reduced churn and made code review easier because each story had a crisp job.

---

## What Didn’t Go Well

### Tracker hygiene lagged behind implementation more than once
Story 2.2 carried a review finding for tracker contradiction, and Story 2.3 also needed tracker synchronization at close-out. Nothing here was product-risky, but it creates avoidable confusion about the real state of work.

### Some runtime edge cases were discovered later than ideal
Non-ASCII token fail-closed behavior, IPv6 issuer host formatting, and cleanup behavior around smoke subprocesses surfaced during review rather than in the first implementation pass. They were fixed, but the pattern says boundary-sensitive runtime code still benefits from more aggressive edge-case thinking earlier.

### The “strict no-op” contract is still harder than it looks
Epic 1 already warned about this, and Epic 2 confirmed it. Preserving `AUTH_MODE=none` and stdio cleanliness while introducing real auth wiring required explicit tests, guarded imports, and review attention. That is not a one-time solved problem; it is an ongoing regression risk.

### Convenience tooling is useful, but it needs careful framing
`make test-auth` and `make smoke-auth` are helpful focused commands, but they are not substitutes for broader regression validation. The epic handled this correctly in practice, but the distinction needs to stay explicit going into Epic 3 where the surface gets wider.

---

## Lessons Learned

1. The fastest path for auth work is still narrow, story-scoped delivery with hard boundaries between verifier logic, server wiring, and regression expansion.
2. Disabled-auth and stdio behavior are not “background assumptions.” They are first-class acceptance criteria every time auth work expands.
3. Review findings on auth code tend to cluster around fail-closed behavior, cleanup, and boundary leakage. Those should be treated as standard pre-review check areas in future stories.
4. Tracker correctness is part of engineering quality. If the implementation is done but the story state is wrong, the workflow signal is degraded for every later step.
5. Epic 2 validated the static path without changing the larger Epic 3 architecture direction. That is a good sign: the plan appears to be unfolding in a stable sequence rather than requiring a redesign.

## Next Epic Preparation

Epic 3 appears ready to start, and Epic 2 did not reveal a fundamental planning flaw in the OIDC direction.

What Epic 2 prepared successfully for Epic 3:
- the `TokenVerifier` factory and auth module boundary already exist
- FastMCP `AuthSettings` and HTTP auth wiring are already proven in the static path
- disabled-auth and stdio guardrails have live regression coverage
- focused auth test commands now exist for faster iteration on verifier and smoke behavior

What Epic 3 must preserve:
- keep auth logic server-local and verifier-local
- keep preflight as the operator-facing startup validation boundary
- preserve fail-closed semantics and no-secret leakage
- keep `AUTH_MODE=none` and stdio behavior untouched

Recommended preparation focus before or during early Epic 3 work:
- codify the remaining validation-ownership guidance more explicitly so OIDC preflight and verifier responsibilities stay crisp
- start Epic 3 with Story `3-1-implement-oidc-jwt-verifier-with-cached-jwks`
- keep OIDC unit and smoke design free of external provider dependencies, matching the static path’s discipline

No retrospective evidence suggests Epic 3 needs replanning before Story 3.1 starts.
