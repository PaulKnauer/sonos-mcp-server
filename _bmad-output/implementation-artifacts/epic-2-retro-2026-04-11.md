## What Went Well

### Capability-Module Pattern Held
Story 2.1 slotted `input_service.py` + `tools/inputs.py` in without any structural debate. The architecture absorbed the new capability cleanly — the module naming convention paid off.

### Topology Snapshot Pattern (Story 2.2)
`_resolve_group_snapshot()` loads one discovery snapshot per service call and derives room, coordinator, and member list from it. This is the right approach. Established cleanly in Story 2.2 after double-discovery was flagged in Story 2.1's review.

### Deterministic Pre-Mutation Validation (Story 2.3)
`group_rooms()` validates empty set, duplicates, unknown names, and coordinator-not-in-set before any adapter mutation. This is the gold standard for guard implementation in this project.

### Documentation Was In-Epic
Story 2.4 was planned from the start as the hardening/docs story and delivered on time. Integration guides (Home Assistant, n8n), README, and prompt examples all updated within the epic.

### Regression Discipline Maintained
974 → 1119 tests across 4 stories, zero regressions at any step. `make test` before review is holding.

## What Didn't Go Well

### Double-Discovery Bug (Story 2.1)
`_ensure_support` called `get_room` after an already-resolved speaker lookup, producing two redundant discovery calls. Found in review. Story 2.2 fixed this by design with the snapshot pattern — but it took a review finding to establish the right approach.

**Root cause:** No explicit "one topology snapshot per service operation" rule in the architecture doc.

### Post-Implementation Follow-Up Fixes (Story 2.2)
Two follow-up fixes needed after initial implementation:
- Case-insensitive room lookup missing in group-audio operations
- Negative absolute and relative group-volume values not rejected before adapter call

Both are pre-adapter validation items. The pattern from Story 2.3 (deterministic guard before any mutation) should have been applied from the start.

### Epic 1 Action Items Not Tracked
AI-1 and AI-2 from the Epic 1 retro were not actioned before Epic 2 began because they had no tracked home. Retro docs are not surfaced during sprint planning. This was resolved during this retrospective by actioning both items directly and establishing action item tracking as an explicit process requirement.

### Home Assistant Example Error (Story 2.4)
The initial `set_group_volume` example in the Home Assistant integration guide called the tool without first establishing a group — a documentation error that could mislead integrators. Caught in review and fixed.

---

## Lessons Learned

1. **One topology snapshot per service operation is the team rule.** Now documented in `architecture.md` under Communication Patterns. Services should load rooms once, derive coordinator and members from the same snapshot, and not call `list_rooms()` or discovery methods again within the same operation.

2. **Deterministic pre-mutation validation is the gold standard.** Story 2.3's `group_rooms()` implementation is the reference. All guard checks — empty set, unknown names, invalid target state — must fail before the first adapter call.

3. **Retro action items without sprint tracking don't happen.** AI-1 and AI-2 from Epic 1 were aspirational. Going forward, process-improvement actions that require code or doc changes must be folded into the next story that touches the relevant files, or added as explicit sprint-status backlog items.

4. **Documentation errors in examples can mislead integrators.** The Home Assistant `set_group_volume` example was wrong in a way that would produce a runtime error for anyone following it. Docs need the same validation scrutiny as code.
